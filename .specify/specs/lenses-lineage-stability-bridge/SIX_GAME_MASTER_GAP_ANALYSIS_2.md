# Six Game Master Gap Analysis 2: Remaining Bridge Spec Risks

## Verdict

The bridge spec is now pointed in the right direction, but a few implementation traps remain. These are not philosophical issues anymore; they are precision issues.

The remaining danger is that the implementation could technically satisfy the words of the spec while still creating brittle identity, rank, or snapshot behavior.

## Shaman Gap

**Protects:** continuity, memory, living trace.

**What improved:**
- The spec now distinguishes live lineage from snapshot fallback.
- It requires lineage traces to label source.

**Remaining gap:**
- Snapshot timing is still ambiguous. A task snapshot created at attach time and a BAR snapshot copied at plant time could represent different moments. The spec does not say whether BARs should carry the original attach snapshot or a fresh plant-time snapshot.

**Risk:**
- A BAR may feel like it carries the current living thread but actually carries stale historical context from when the task was first attached.

**Close the gap:**
- Define two snapshot moments:
  - `attachSnapshot`: immutable trace when a TTV task is attached to a goal.
  - `plantSnapshot`: immutable trace when that task becomes a BAR.
- Live lineage wins in UI when resolvable, but snapshots should name their moment.

**Gate:**
- A planted BAR can show whether its preserved snapshot was captured at task attach or BAR plant.

## Challenger Gap

**Protects:** sharp constraints, honest limits, no fake challenge.

**What improved:**
- The spec now requires a DB-level uniqueness invariant for task ranks.

**Remaining gap:**
- The rank invariant does not decide whether ranks are active-only or historical. The suggested partial index says “priorityRank IS NOT NULL,” which prevents rank reuse forever, not just among active tasks.

**Risk:**
- Implementation may block legitimate task replacement, or it may claim “active-only” while enforcing historical immutability.

**Close the gap:**
- Pick the rule:
  - **Recommended:** ranks are historical daily keep order; do not reuse them. The visible active cap is separate.
  - Active count is enforced transactionally by status filter.
  - Unique index is `(dailySessionId, priorityRank) WHERE priorityRank IS NOT NULL`.

**Gate:**
- Completing or composting task #2 does not cause a new task to reuse rank #2; it gets the next rank while the active cap still prevents more than five live tasks.

## Regent Gap

**Protects:** law, state, governance.

**What improved:**
- The spec has explicit removed-goal states.
- It says referenced goals are never hard-deleted.

**Remaining gap:**
- It lacks allowed state transitions.
- It lacks a precise definition of “referenced.”

**Risk:**
- `archived`, `parked`, `superseded`, and `complete` become labels without law. Different actions may move goals between states inconsistently.

**Close the gap:**
- Add transition table:
  - `active -> parked | superseded | complete | archived`
  - `parked -> active | archived`
  - `superseded -> archived`
  - `complete -> active | archived`
  - `archived` is terminal except admin/manual restore.
- Define referenced as any of:
  - child `LensGoal.parentGoalId`,
  - `TapTheVeinTask.lensGoalId`,
  - `CustomBar.lensGoalId`,
  - `LensWorkshopDraft.parentGoalId`,
  - `LensGoal.supersededById`,
  - future explicit lineage tables.

**Gate:**
- Every state-changing action validates against the transition table.

## Architect Gap

**Protects:** identity, schema fit, migration feasibility.

**What improved:**
- Stable key is now the identity strategy.
- Title/domain/cadence matching is disallowed for normal edits.

**Remaining gap:**
- `stableKey String @unique` is too specific without a key-generation contract.
- Server-generated stable keys conflict with fully client-side unsaved option editing unless temporary client keys are allowed.

**Risk:**
- Migration fails on duplicate backfill keys, or implementation invents unstable client-only keys that are lost before save.

**Close the gap:**
- Define key policy:
  - Client may create `tempKey` for unsaved options.
  - Server creates canonical `stableKey` on first persistence.
  - Server returns canonical keys to client after save.
  - `stableKey` is globally unique only if generated by CUID/UUID.
  - If keys are scoped, use compound uniqueness `(playerId, stableKey)`.
- Recommended: use CUID/UUID global stable keys and document it.

**Gate:**
- Two unsaved same-title options can be edited locally, saved, and returned with distinct canonical stable keys.

## Diplomat Gap

**Protects:** player trust, clarity, humane recovery.

**What improved:**
- The spec names player-facing copy for states.
- Park/reactivate is required.

**Remaining gap:**
- Tasks mark product decisions complete, but the choices are not summarized in the tasks file.
- Stale lineage copy exists for one failure case, but not for snapshot/history labels.

**Risk:**
- Future implementers read the checklist, miss the decision nuance, and re-open already-settled questions.

**Close the gap:**
- Add a “Resolved Decisions” section to tasks or spec:
  - removed referenced goal behavior,
  - stable key generation,
  - compact vs expanded lineage display,
  - rank semantics,
  - snapshot timing.

**Gate:**
- Someone can read `tasks.md` alone and know the chosen direction.

## Sage Gap

**Protects:** perspective, trace meaning, non-confusion.

**What improved:**
- The lineage viewer now has minimum fields.
- The trace uses “serves” language.

**Remaining gap:**
- It does not yet distinguish living trace, attach-time memory, and plant-time memory.

**Risk:**
- The lineage viewer can show technically correct data without helping the player understand what kind of truth they are seeing.

**Close the gap:**
- Add lineage source vocabulary:
  - `live`: current goal chain,
  - `attach_snapshot`: task’s historical attachment moment,
  - `plant_snapshot`: BAR’s planting moment.
- UI copy:
  - live: “Serving the current goal thread”
  - attach snapshot: “Carrying the thread as it was when attached”
  - plant snapshot: “Carrying the thread as it was when planted”

**Gate:**
- A player can tell whether the displayed lineage is current or historical.

## Priority Spec Patches

1. **Stable key generation contract**
   - CUID/UUID global stable keys, or compound uniqueness.
   - Temporary client keys allowed only before first persistence.

2. **Rank semantics**
   - Decide historical keep order vs active-only rank.
   - Recommended: historical rank immutability plus separate active cap.

3. **State transition table**
   - Define allowed transitions and terminal states.

4. **Referenced-goal definition**
   - Define exact records that make a goal non-deletable.

5. **Snapshot timing vocabulary**
   - `live`, `attach_snapshot`, `plant_snapshot`.

6. **Resolved decisions section**
   - Make tasks/spec self-contained for implementers.

## Recommendation

Patch the spec one more time before implementation. These changes are small but load-bearing. They prevent the next coding pass from making a reasonable-but-wrong choice in the places where lineage stability depends on precision.
