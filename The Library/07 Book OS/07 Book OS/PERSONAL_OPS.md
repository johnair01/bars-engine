## Spec Routing Flow (Personal vs bars-engine)

**Policy:** Every new spec must be routed to a home immediately after creation.

### Routing Decision (intelligent)
Score each spec quickly:
- `Product/Code dependency` (0-2): Is implementation tied to bars-engine code, APIs, runtime, or experiments?
- `Personal operating behavior` (0-2): Is this about workflow, governance, writing process, decision hygiene, or personal execution?
- `Audience of record` (0-1): Who must reliably reference it first? (`bars-engine builders` vs `Wendell ops`)

**Route rule:**
- If Product score > Personal score -> store in bars-engine ops docs
- If Personal score > Product score -> store in Personal Ops / workspace ops
- If tied -> default to Personal Ops and add cross-post pointer

### Canonical Destinations
- Personal / governance / operating-system specs -> `PERSONAL_OPS.md` + related root specs
- Product implementation / engineering execution specs -> `bars-engine/docs/runbooks/` (or nearest bars-engine ops doc)

### Cross-Post Option (required when split ownership exists)
When a spec impacts both domains:
1. Choose one canonical owner file (single source of truth).
2. Create a short cross-post entry in the secondary domain with:
   - Spec title
   - Canonical path
   - 1-2 line relevance note
   - Last-updated date
3. Never maintain two divergent full specs.

### Intake Checklist (apply to every new spec)
- `Owner:` Personal Ops or bars-engine ops
- `Canonical path:` exact file path
- `Cross-post needed:` yes/no
- `If yes:` add pointer entry same session
