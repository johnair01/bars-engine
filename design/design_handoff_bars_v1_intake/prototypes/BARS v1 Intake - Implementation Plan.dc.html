The gap · what v1 actually changes

Five gaps between today's form and the design

The current CreateBarFormPage persists text, photos and links — but it's a plain zinc/purple form, captures don't get a maturity phase, there's no provenance ritual, and there's no Tune gate. Here's the delta, each tagged IN or FAST-FOLLOW.

G1

Aesthetic — it doesn't feel like the world yet

Today: black + zinc-700 borders + purple-600 buttons, generic. Target: the OS-contains-cards surface — element-tinted field, inset highlight, provenance strip, "Keep · tune later". This is the delight that drives dogfooding.

IN · v1
G2

Maturity isn't stamped at capture

createPlayerBar never writes seedMetabolization, so new captures have a null maturity and fall out of the garden / gate logic. Fix: set maturity:'captured' on create.

IN · v1
G3

No provenance ritual

The design locks time / place / sensation at the moment of capture. v1 stamps the cheap, immutable parts — createdAt + a derived time-of-day phrase ("dusk"). Place & sensation are deferred.

IN · v1
G4

No Tune gate

The channel columns exist but nothing writes them, and there's no screen to assign element / altitude / move and advance maturity. Build /bars/[id]/tune + a tuneBar() action.

IN · v1
G5

Stories-rich canvas (drag / recolor / share)

The draggable-text composer + share-to-Stories / video / deck needs a new captureDesign JSON column and an export pipeline. Real delight, but more than a day — it's the fast-follow that builds on a shipped capture loop.

Create contract · the only server change
// in createPlayerBar() / createBarForUpload() // add to db.customBar.create({ data: {...} }): type: 'charge_capture', // BSM-eligible type nation: fieldTint ?? null, seedMetabolization: JSON.stringify({ maturity: 'captured', soilKind: 'holding_pen', }), // createdAt is auto · provenance locked

Both BSM-eligible types ('bar', 'charge_capture') pass the maturity guards. Use mergeSeedMetabolization() rather than raw JSON if you prefer the helper.

Tune — the gate into the game

Route /bars/[id]/tune. Three channels turn a raw scribble into game data. Each assignment is one server write that also advances maturity. The card preview responds live so the alchemy is visible.

Maturity only ever moves forward — clamp to the highest phase reached. 'integrated' stays reserved for graduation via growQuestFromBar().

New action · tuneBar()
// src/actions/bars.ts — owner-guarded, BSM-typed export async function tuneBar(barId, patch: { nation?: string intensity?: string // altitude emotionalAlchemyTag?: string moveType?: string }) { // 1 · auth: bars_player_id cookie // 2 · guard: creator-only + type ∈ {bar,charge_capture} // 3 · db.customBar.update({ data: patch }) // 4 · derive + updateBarSeedMaturity(barId, next) // 5 · revalidate /bars, /bars/[id], /bars/garden }
Acceptance · Tune
◇Picking an element re-tints the live card to that nation.
◇Each channel persists immediately and survives reload.
◇Maturity advances and never regresses.
◇Altitude sets the card's data-altitude glow.
◇A fully-tuned BAR shows a "ready" / graduate affordance.
◇Tuning is optional for keeping — only required to play.

Definition of done · the dogfood test

You have arrived successfully if, by end of day, you captured one charge in under ten seconds — and tuned one BAR into game data.

01Open /bars/create on your phone, type a line, Keep. It feels good.
02The card is on your board with locked provenance and a captured-phase dot.
03Open it, tune element + altitude + move; watch it glow up to game-ready.
04No migration ran, no existing flow regressed, and you want to capture again.

That last line is the whole point. The yardstick isn't "the feature works" — it's "you reach for it without thinking." Steady accumulation is the form; this sprint paves the first brick.
